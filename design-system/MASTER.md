# React Admin UI/UX 主规范

本文档是本项目后台管理系统前端的 UI/UX 全局约束。后续开发页面、布局、公共组件和样式时，默认以本文档为准。

如果某个页面确实需要特殊规则，应把覆盖说明写到 `design-system/pages/` 下。

## 1. 整体设计定位

本项目定位为企业后台管理系统，不是营销官网。

核心原则：

- 优先保证信息密度、可读性和操作效率。
- 使用稳定、可预测、易复用的后台布局。
- 视觉风格保持简洁、专业、克制。
- 少用装饰，不使用大面积渐变、大插画、大留白、夸张动画。
- 页面优先复用公共组件，不为每个页面单独发明样式。
- 让数据扫描、筛选、对比、批量操作变得高效。
- 组件可组合优先于页面定制。

## 2. 页面结构规范

后台统一使用稳定的管理端骨架：

- `AppShell`：侧边栏、顶部栏、主内容区。
- `PageHeader`：面包屑、页面标题、简短说明、主操作。
- `Toolbar`：页面级工具区，可承载搜索、筛选、批量操作、视图切换。
- `MainContent`：表格、Dashboard 网格、表单分组、详情分区等核心内容。

默认布局规则：

- 桌面端页面内边距：`24px`。
- 平板和移动端页面内边距：`16px`。
- 主内容模块间距：`24px`。
- 大区块间距：`24px` 到 `32px`。
- 侧边栏展开宽度：`240px`。
- 侧边栏收起宽度：`64px`。
- 顶部栏高度：`56px` 到 `64px`。
- 列表页允许使用全宽内容区，便于展示更多列。
- 表单页正文宽度通常限制在 `960px` 到 `1200px`。
- 主内容区只承载当前页面的核心任务，避免混入过多无关模块。

## 3. 色彩体系

采用中性的蓝灰企业后台色彩体系。

基础色：

| Token | 值 | 用途 |
| --- | --- | --- |
| `color.primary` | `#1677FF` | 主按钮、链接、选中态、激活导航 |
| `color.primaryHover` | `#155EEF` | 主色悬停态 |
| `color.textPrimary` | `#1F2937` | 主要文字 |
| `color.textSecondary` | `#4B5563` | 次要文字 |
| `color.textTertiary` | `#6B7280` | 说明、提示、元信息 |
| `color.border` | `#E5E7EB` | 分割线、输入框边框、卡片边框 |
| `color.background` | `#F5F7FA` | 页面背景 |
| `color.surface` | `#FFFFFF` | 卡片、表格、表单、弹窗背景 |

语义色：

| Token | 值 | 用途 |
| --- | --- | --- |
| `color.success` | `#16A34A` | 成功状态 |
| `color.warning` | `#D97706` | 警告状态 |
| `color.error` | `#DC2626` | 错误、危险操作 |
| `color.info` | `#0891B2` | 信息提示 |

使用规则：

- 主色只用于可操作重点，不大面积铺满页面。
- 状态色只表达状态，不作为装饰色滥用。
- 图表颜色通常不超过 6 组。
- 避免紫色主导、深蓝主导、米色主导、营销式渐变主题。
- 页面内禁止随手写新颜色，必须先判断是否应进入 token。
- shadcn/ui 的主题变量应映射到本项目 token，而不是直接采用默认视觉。

## 4. 字体层级

推荐字体栈：

~~~css
font-family: Inter, "PingFang SC", "Microsoft YaHei", "Noto Sans SC", sans-serif;
~~~

数字和表格可启用等宽数字：

~~~css
font-variant-numeric: tabular-nums;
~~~

字号层级：

| 场景 | 字号 / 行高 | 字重 |
| --- | --- | --- |
| 页面标题 | `24px / 32px` | `600` |
| 区块标题 | `18px / 28px` | `600` |
| 卡片标题 | `16px / 24px` | `600` |
| 正文 | `14px / 22px` | `400` |
| 表单标签 | `14px / 22px` | `500` |
| 次要文字 | `13px / 20px` | `400` |
| 辅助说明 | `12px / 18px` | `400` |
| 表格单元格 | `14px / 22px` | `400` |

字体规则：

- 后台系统默认正文使用 `14px`。
- 不要把普通正文降到 `12px`。
- 不使用随视口宽度变化的字号。
- 默认字间距为 `0`，不要使用负字间距。
- 数据、金额、编号、状态码等内容优先启用 `tabular-nums`。

## 5. 间距体系

使用 `4px` 基准栅格。

推荐间距 token：

| Token | 值 | 用途 |
| --- | --- | --- |
| `space.1` | `4px` | 图标和文字的紧凑间距 |
| `space.2` | `8px` | 紧凑组件间距 |
| `space.3` | `12px` | 表单内部间距 |
| `space.4` | `16px` | 卡片内边距、工具栏间距 |
| `space.5` | `20px` | 高密度卡片内边距 |
| `space.6` | `24px` | 页面和模块间距 |
| `space.8` | `32px` | 大区块间距 |

密度规则：

- 表格行高：紧凑 `40px`，默认 `44px`，舒适 `48px`。
- 按钮高度：小 `32px`，默认 `36px`，大 `40px`。
- 输入框高度要与按钮高度体系对齐。
- 不为单个页面临时发明间距值。
- 同级模块使用一致间距，不按视觉感觉随意调整。

## 6. 组件统一样式规范

### 卡片

- 背景使用 `color.surface`。
- 边框使用 `1px solid color.border`。
- 圆角使用 `10px` 或 `12px`。
- 内边距使用 `16px` 或 `20px`。
- 阴影保持非常轻，默认可以不用阴影。
- 卡片头部、内容区、底部操作区应分层明确。
- 禁止无意义的卡片套卡片。

### 按钮

- 主按钮使用主色实底。
- 次按钮使用中性描边或浅色背景。
- 危险按钮使用错误语义色。
- 圆角使用 `8px`。
- 高度使用 `32px`、`36px` 或 `40px`。
- 常见工具按钮可优先使用图标，但必须保证含义清晰。
- 所有可点击元素必须有 hover、active、disabled、focus 状态。
- 不使用过重阴影和夸张 hover 动画。

### 表格

- 列表和管理类页面默认优先使用表格。
- 表头需要有明确分隔，可用浅背景或底部分割线。
- 行 hover 状态要清晰但克制。
- 操作列通常固定在右侧。
- 长文本默认截断，并提供 tooltip 或详情入口。
- 必须考虑加载、空数据、错误、分页、选中行状态。
- 窄屏下使用横向滚动或专门的卡片布局，不能让表格撑破视口。
- 第一阶段可使用基础表格，后续复杂表格能力再基于公共 `DataTable` 增强。

### 表单

- 默认使用顶部标签，方便阅读和响应式适配。
- 必填字段必须有统一标识。
- 大多数字段在 blur 时校验，提交时做整体校验。
- 错误提示固定展示在字段下方，避免布局跳动。
- 根据字段类型使用正确的 `type`、`inputMode`、`autoComplete`。
- 提交时必须有 loading、成功、失败反馈。
- 搜索筛选表单与编辑表单应区分样式和结构，不混用。

### 弹窗和抽屉

- 弹窗用于确认、短流程、小编辑。
- 抽屉适合从列表或详情页进入上下文编辑。
- 长表单使用独立页面，不塞进弹窗。
- 常用弹窗宽度：`480px`、`640px`、`800px`。
- 标题区、正文区、底部操作区保持统一。
- 危险确认必须明确表达影响。
- 第一阶段不强制实现弹窗和抽屉组件，按页面需求逐步补充。

## 7. 页面推荐布局模式

### Dashboard

推荐顺序：

- 顶部 KPI 卡片。
- 中部趋势图和关键分布/对比区域。
- 底部业务模块：待办、告警、最近操作、异常记录。

规则：

- 一屏内优先回答：当前状态如何、哪里异常、下一步要做什么。
- 第一阶段可以使用图表占位，不强制引入复杂图表库。
- 正式实现图表时，应保证图表颜色克制、数据可读，并提供必要的文字说明。

### 列表页

推荐结构：

- `PageHeader`
- `SearchFilterBar`
- `TableToolbar`
- `DataTable`
- `Pagination`

列表页规则：

- 搜索和筛选靠近表格。
- 批量操作放在表格上方。
- 新建、导入等主操作放在 `PageHeader`。
- 筛选条件尽量支持保留状态。
- 默认考虑加载态、空态、错误态和分页。
- 复杂筛选不要挤占表格主体空间，可折叠或收纳。

### 表单页

推荐结构：

- `PageHeader`
- `FormSection`
- 固定或清晰分隔的提交操作区。

表单页规则：

- 默认单栏。
- 只有字段短且强相关时才使用双栏。
- 复杂表单按业务含义分组。
- 长表单可以使用锚点导航。
- 提交、重置、返回等动作位置应稳定可预测。

### 详情页

推荐结构：

- 顶部摘要区，包含状态和主操作。
- 关键元信息区。
- 业务详情分区。
- 日志、时间线、关联资源分区。

详情页规则：

- 重要状态和下一步操作必须靠前展示。
- 只读字段使用统一的 label/value 对齐方式。
- 除非内容非常多，否则不要把关键内容藏到 tab 里。

## 8. 公共组件建设优先级

第一阶段优先建设：

1. `AppShell`
2. `AppHeader`
3. `AppSidebar`
4. `PageHeader`
5. `ContentCard`
6. `StatusTag`
7. `EmptyState`
8. `RequireAuth`

第二阶段根据页面需求逐步建设：

1. `SearchFilterBar`
2. `TableToolbar`
3. `DataTable`
4. `FormSection`
5. `Breadcrumbs`
6. `ActionBar`
7. `ConfirmDialog`
8. `AsyncButton`
9. `StatisticCard`
10. `DetailSection`

后续复杂场景再考虑：

1. `ModalForm`
2. `DrawerForm`
3. `PermissionGuard`

组件建设规则：

- 优先扩展公共组件参数，不复制组件代码做页面定制。
- 新增组件前，先判断是否能并入已有组件变体体系。
- 不为了预想中的未来需求提前做复杂封装。
- 第一阶段以可读、稳定、可复用为主。

## 9. React 和可访问性规则

React 规则：

- 使用语义化 HTML：点击操作用 `button`，导航用 `nav`，表单用 `form`。
- 可编辑表单使用受控输入。
- 搜索和复杂筛选使用 `useDeferredValue` 或 debounce，避免每次输入都触发重计算。
- 不在页面里重复写大量样式，能沉淀到组件或 token 就沉淀。
- 状态优先放在使用它的最近位置，确实跨模块共享时再上提。
- 不用 `div` 模拟按钮、链接、输入框等原生交互元素。

可访问性规则：

- 键盘 Tab 顺序要符合视觉顺序。
- 所有交互元素必须有清晰 focus 状态。
- `AppShell` 建议提供跳转到主内容的 skip link。
- 不允许移除 outline 后没有替代 focus 样式。
- 不能只靠颜色表达状态。
- 有意义的图片必须提供 alt。
- 动画需要尊重 `prefers-reduced-motion`。

## 10. 后续逐页开发流程

开发任何新页面前，先按这个流程走：

1. 判断页面类型：`dashboard`、`list`、`form`、`detail`、`settings`、`exception`，或者记录为例外。
2. 查看 `design-system/pages/` 下是否存在页面覆盖规则。
3. 使用本文档里的对应布局模式。
4. 优先复用已有公共组件。
5. 只有重复出现的样式需求才新增 token。
6. 补齐加载、空数据、错误、禁用等基础状态。
7. 如页面涉及权限，再补齐权限状态；第一阶段不实现 RBAC。
8. 检查桌面、平板、移动端基本表现。
9. 页面专属 CSS 保持最少，可复用样式沉淀到组件。

硬性约束：

- 后台功能页不做营销官网布局。
- UI 图标不使用 emoji。
- 操作型页面里不使用超大 hero 字体。
- 页面里不随手写 token 之外的颜色。
- 禁止无意义的卡片套卡片。
- 禁止表格撑破视口。
- 禁止键盘 focus 不可见。
- 不实现需求外的复杂功能。

# AGENTS.md

## 角色

你是本项目的前端开发助手，负责协助开发一个基于 React 的后台管理系统前端模板。

你的核心任务是：在不破坏现有结构和规范的前提下，帮助用户实现清晰、稳定、可维护、可扩展的后台管理系统前端代码。

## 交流要求

- 始终使用中文与用户交流。
- 解释、总结、计划和说明使用中文。
- 代码命名、文件命名、类型命名、变量命名可使用英文。
- 不主动切换为英文。
- 回答应清晰、直接、克制，不输出无关扩展内容。

## 项目定位

本项目是一个 React Admin 前端基础模板，用于后续复用到不同后台管理系统中。

第一目标是保持：

- 结构清晰
- 风格统一
- 易于扩展
- 易于维护
- 不过度设计

本项目当前阶段优先实现后台管理系统基础骨架和常见页面类型，不追求复杂业务能力。

## 技术方向

默认采用以下技术方向：

- React
- TypeScript
- Vite
- Tailwind CSS
- shadcn/ui
- React Router
- Zustand

如项目实际依赖与以上不一致，以项目现有代码和用户最新要求为准。

## 包管理与命令约定

默认使用项目当前已有的包管理器和 lock 文件，不要擅自切换包管理器。

如果项目中存在：

- `package-lock.json`：优先使用 `npm`
- `pnpm-lock.yaml`：优先使用 `pnpm`
- `yarn.lock`：优先使用 `yarn`

不要在未确认的情况下混用多个包管理器。

完成修改后，优先执行项目已有验证命令，例如：

```bash
npm run build
```

如果项目中存在以下命令，也应按需执行：

```bash
npm run lint
npm run typecheck
```

具体以 `package.json` 中已有 scripts 为准。

## UI/UX 规范

React Admin UI/UX 主规范位于：

```text
design-system/MASTER.md
```

开发任何页面、布局、组件和样式时，必须优先遵循该规范。

尤其需要遵守：

- 后台管理系统风格，不做营销官网风格。
- 保持中性、专业、高密度、弱装饰。
- 页面结构、间距、配色、圆角、阴影、字体层级应保持统一。
- 优先使用 design tokens，不在页面中随意写死颜色、间距、圆角、阴影。
- shadcn/ui 只作为基础组件来源，最终视觉以 `design-system/MASTER.md` 为准。
- 信息优先于视觉表现。
- 操作效率优先于炫技效果。
- 避免大面积渐变、大插画、大留白和过度动画。

## 开发原则

- 先理解现有结构，再修改代码。
- 修改前应查看相关目录、已有组件、已有样式和已有类型。
- 优先复用已有组件。
- 新增组件前，先判断是否能扩展已有组件。
- 保持组件职责单一。
- 避免过度封装。
- 避免一次性引入复杂功能。
- 不随意改变既有目录结构和公共 API。
- 不实现需求外的功能。
- 不凭空新建与现有组件职责重复的组件。
- 不为了少量页面需求提前设计复杂框架。
- 不主动重构无关代码。

## 页面开发规则

开发新页面时，应先判断页面类型：

- Dashboard
- List
- Form
- Detail
- Settings
- Exception

然后选择对应的页面骨架和公共组件。

同类页面应保持一致的结构、间距和交互方式。

页面内容区应优先使用：

- `PageHeader`
- `ContentCard`
- `SearchFilterBar`
- `TableToolbar`
- `DataTable`
- `FormSection`
- `StatusTag`
- `EmptyState`

如现有组件不能满足需求，应优先小幅扩展，而不是重新发明一套页面结构。

## shadcn/ui 使用规则

- 可以使用 shadcn/ui 作为基础组件来源。
- 不直接照搬默认视觉风格。
- 应基于项目设计规范进行样式适配。
- 页面级功能应优先封装为业务组件，而不是直接堆砌基础组件。
- 不因为 shadcn/ui 提供某组件就盲目引入。
- 新增 shadcn/ui 组件前，应确认确实有复用价值。

## 状态管理规则

- 全局状态优先使用 Zustand。
- 只将真正跨页面、跨组件共享的状态放入 store。
- 页面局部状态应保留在页面或组件内部。
- 不要把所有表单状态、弹窗状态、临时筛选状态都放入全局 store。

## 数据访问规则

当前阶段默认不接真实后端。

已有数据访问分层时，应遵循：

- `src/api`：页面调用的数据访问入口。
- `src/mocks`：本地 mock 数据和 mock 行为。
- `src/types`：领域类型、接口类型和通用类型。

页面不应直接堆大量 mock 数据。

如果需要新增 mock 数据，应优先放入 `src/mocks`，再通过 `src/api` 暴露给页面使用。

## 表单开发规则

表单优先使用项目现有方案。

如项目已使用：

- `react-hook-form`
- `zod`
- `@hookform/resolvers`

则继续沿用该方案。

表单校验规则应集中定义，避免在 JSX 中散落大量校验逻辑。

基础输入组件应保持通用，不应绑定具体业务字段。

## 路由与权限规则

第一版只做登录态守卫，不做复杂权限系统。

默认边界：

- 不实现 RBAC。
- 不实现动态路由。
- 不实现动态菜单接口。
- 不实现复杂权限指令。
- 不实现多租户权限模型。

后台页面应通过 `RequireAuth` 或现有守卫机制保护。

未登录访问后台页面时，应跳转到登录页。

## 禁止主动引入的能力

除非用户明确要求，不要主动引入以下能力：

- 真实后端接口
- RBAC 权限系统
- 动态菜单
- 多标签页 Tabs
- 暗色模式
- 国际化
- 复杂主题编辑器
- 复杂表格配置器
- 大型图表库
- 大型状态管理替代方案
- Ant Design、MUI 等新的 UI 框架
- TanStack Query 等新的数据请求框架
- 微前端
- SSR / SSG
- 单元测试框架
- E2E 测试框架

如确实认为需要新增依赖，必须先说明原因、收益和影响范围。

## 代码质量

- 使用 TypeScript。
- 保持命名清晰。
- 保持文件职责清晰。
- 删除无用代码。
- 避免重复逻辑。
- 避免引入不必要依赖。
- 避免过深嵌套。
- 避免超大组件。
- 组件 props 应尽量明确、稳定。
- 公共类型应放在合适的 `types` 文件中。
- 不要为了抽象而抽象。

## 样式规则

- 优先使用 Tailwind CSS 和项目 design tokens。
- 不在页面中随意写死大量颜色、间距、圆角和阴影。
- 页面背景、卡片、边框、阴影、字体层级应保持统一。
- 表格、表单、筛选区、操作按钮应保持后台系统的信息密度。
- 不做营销站式视觉效果。
- 不添加无意义动画。
- 不引入复杂主题系统。

## 修改前检查清单

执行较大改动前，应先检查：

- 当前目录结构
- 相关页面实现
- 相关公共组件
- 相关类型定义
- 相关 mock 数据
- 相关 API 封装
- `design-system/MASTER.md`
- `package.json` scripts

不要只根据文件名猜测实现。

## 执行习惯

在执行较大改动前，先简要说明计划。

计划应包含：

- 准备修改哪些部分
- 为什么这样改
- 是否会影响现有结构
- 如何验证

小改动可以直接执行，但完成后仍需说明修改内容。

## 完成后总结

每次完成修改后，应总结：

- 修改了哪些文件
- 实现了什么
- 如何运行或验证
- 是否存在未完成事项
- 是否引入了新依赖

如执行了验证命令，应说明验证结果。

## 当前阶段边界

当前阶段重点是：

- 能登录
- 能进入后台
- 有稳定布局
- 有基础路由
- 有示例页面
- 有统一风格
- 有可复用公共组件
- 后续方便扩展

当前阶段不追求：

- 大而全
- 复杂权限
- 复杂业务
- 复杂工程化
- 复杂视觉系统
- 复杂后端集成
